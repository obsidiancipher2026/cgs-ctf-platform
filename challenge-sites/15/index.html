Insecure Deserialization — RCE

Exploit a PHP unserialize() vulnerability to execute arbitrary code via a crafted object.

Hard 160 points

Challenge Overview

A PHP web application uses unserialize() on user-controllable input. This allows an attacker to inject arbitrary PHP objects. If a class defines magic methods like __destruct or __wakeup, they are automatically invoked during deserialization.

How Insecure Deserialization Works

PHP's unserialize() reconstructs a serialized string back into a PHP object. During this process:

  • unserialize() reconstructs the object from the string
  • When the object is destroyed (end of request), __destruct() is called automatically
  • If __destruct() performs dangerous operations (like reading files or executing commands), an attacker can control its behavior by setting object properties

This is known as a PHP Object Injection or Insecure Deserialization vulnerability.

Deserialization Lab

The application accepts serialized data via the data cookie. The server calls unserialize(base64_decode($_COOKIE['data'])).

PHP Class Definition

<?php class Log { public $file = 'default.log'; public function __destruct() { echo "[Log] Reading: " . $this->file; if (file_exists($this->file)) { echo file_get_contents($this->file); } } } $log = unserialize(base64_decode($_COOKIE['data']));

Serialized Format

O:3:"Log":1:{s:4:"file";s:18:"/etc/passwd";} // O:3:"Log" — Object of class "Log" (3 char name) // :1: — 1 property // s:4:"file" — String property name (4 chars) // s:18:"..." — String value (18 chars)

Send Serialized Payload

Either field is accepted. The server will decode base64 if needed.

Hints

  • Certain PHP functions reconstruct data structures from string representations
  • Objects may contain methods that are automatically invoked during certain lifecycle events
  • The timing of these automatic invocations follows specific rules
  • Properties set during reconstruction influence the behavior of these automatic methods

Key Observations

  • Line 27: unserialize() is called directly on user input (only base64 decoded)
  • Line 6-16: The Log class has a __destruct() method that reads any file
  • Gadget chain: Attacker controls $file property → __destruct() reads any file on the server
  • Impact: Arbitrary file read, potentially leading to RCE (e.g., overwriting PHP files or reading sensitive credentials)
